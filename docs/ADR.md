# Architecture Decision Records

## 철학
MVP 속도 최우선. 사용자 1명을 위한 도구. 작동하는 최소 구현을 선택하고 오버엔지니어링을 금지한다.

---

### ADR-001: 데이터 소스로 서울 버스도착정보 API 채택
**결정**: 카카오버스 API 대신 서울 버스도착정보 API 사용.
**이유**: 카카오버스는 공개 API가 없다. 서울 버스도착정보 API는 ARS ID 기준 도착정보 + 혼잡도를 제공한다.
**트레이드오프**: 서울 외 지역 버스는 지원하지 않는다 (MVP 범위 = 서울).

### ADR-002: 출발 정류장만 입력, 도착 정류장 입력 없음
**결정**: stops.json에 출발 정류장 ARS ID + 타는 버스 번호(routes)만 둔다. 도착지는 label 텍스트로만.
**이유**: 도착시간·혼잡도는 출발 정류장 기준으로 측정되므로 도착지 ID는 계산에 불필요하다. "특정 목적지 가는 버스"는 (올바른 출발 정류장) + (routes 필터)로 완전히 표현된다.
**트레이드오프**: 봇이 경로를 자동 탐색/검증하지 않는다. 어떤 버스가 목적지로 가는지는 사용자가 직접 알아야 한다 (매일 타므로 이미 안다).

### ADR-003: Python + requests, 프레임워크 최소화
**결정**: 텔레그램은 python-telegram-bot 대신 `requests`로 직접 폴링. 상태 저장 없음.
**이유**: 명령어 2개뿐인 봇에 비동기 프레임워크/DB는 과하다. requests 폴링이 동기·단순·테스트 용이.
**트레이드오프**: 명령 라우팅을 일부 직접 구현. 대규모 확장에는 부적합 (MVP라 무관).

### ADR-004: Railway worker 폴링 배포
**결정**: 웹훅이 아닌 long-polling worker로 배포.
**이유**: 인바운드 포트/도메인/웹훅 설정 없이 프로세스만 띄우면 된다. 가장 단순.
**트레이드오프**: 봇 프로세스가 상시 떠 있어야 한다 (Railway worker).
